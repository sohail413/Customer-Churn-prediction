
# 📉 Customer Churn Prediction

A machine learning project that predicts whether a telecom customer will churn (leave the service) based on their account information, services subscribed, and billing details. Built on the **Telco Customer Churn** dataset from Kaggle.

---

## 📌 Problem Statement

Customer churn is one of the most critical metrics for subscription-based businesses. Acquiring a new customer costs significantly more than retaining an existing one. This project builds a classification model that predicts customer churn so that a business can proactively target at-risk customers with retention offers.

---

## 📊 Dataset

- **Source:** [Telco Customer Churn (Kaggle)](https://www.kaggle.com/datasets/blastchar/telco-customer-churn)
- **Rows:** 7,043 customers
- **Columns:** 21 (20 features + target `Churn`)
- **Target:** `Churn` (Yes/No) — imbalanced, ~26.5% churn rate (1,869 churned vs 5,174 retained)

### Feature groups
| Category | Features |
|---|---|
| Demographics | `gender`, `SeniorCitizen`, `Partner`, `Dependents` |
| Account info | `tenure`, `Contract`, `PaperlessBilling`, `PaymentMethod` |
| Services | `PhoneService`, `MultipleLines`, `InternetService`, `OnlineSecurity`, `OnlineBackup`, `DeviceProtection`, `TechSupport`, `StreamingTV`, `StreamingMovies` |
| Billing | `MonthlyCharges`, `TotalCharges` |

---

## 🔍 Exploratory Data Analysis (EDA)

Key findings from `notebooks/02_eda.ipynb` / `src/eda.py`:

- **`TotalCharges`** had 11 blank string values, all corresponding to customers with `tenure = 0` (brand-new customers with no billing history yet). These were imputed with `0.0`.
- **Tenure** has a bimodal distribution — a large spike of very new customers (tenure ≈ 0) and another spike of long-tenured customers (tenure ≈ 70), suggesting two distinct customer segments: churners-in-waiting and loyal long-termers.
- **`tenure`** and **`TotalCharges`** are strongly correlated (0.83), as expected (longer tenure → more accumulated billing).
- **`MonthlyCharges`** and **`TotalCharges`** are moderately correlated (0.65).
- Customers on **month-to-month contracts** dominate the dataset and are visually associated with higher churn versus one/two-year contracts.
- Class imbalance in the target (73.5% No / 26.5% Yes) motivated the use of **SMOTE** oversampling on the training set only.

Plots generated: histograms + boxplots for numeric features (`tenure`, `MonthlyCharges`, `TotalCharges`), a correlation heatmap, and count plots for every categorical feature.

---

## ⚙️ Preprocessing Pipeline

1. Drop `customerID` (identifier, not predictive).
2. Fix `TotalCharges`: convert blank strings → `0.0`, cast to `float`.
3. Encode target: `Churn` → `{No: 0, Yes: 1}`.
4. Label-encode all remaining categorical columns; encoders are persisted to `models/encoders.pkl` so the exact same encoding can be reapplied at inference time.
5. Train/test split (80/20, `random_state=42`).
6. Apply **SMOTE** to the training split only (never touch the test set) to correct class imbalance.

---

## 🤖 Modeling

Three models were compared using 5-fold cross-validation (accuracy) on the SMOTE-balanced training data:

| Model | CV Accuracy (mean) |
|---|---|
| Decision Tree | ~0.80 |
| **Random Forest** | **~0.84** ✅ |
| XGBoost | ~0.83 |

> Exact numbers will vary by run/environment — see console output when you run `src/train.py`.

**Random Forest** was selected as the final model and evaluated on the held-out test set with accuracy, confusion matrix, and a full classification report (precision/recall/F1 per class).

The trained model + feature name list are saved together to `models/customer_churn.pkl` for reuse at inference time.

## 🚀 How to Run

### 1. Clone and set up environment
```bash
git clone https://github.com/<your-username>/churn-prediction.git
cd churn-prediction
python -m venv venv
source venv/bin/activate   # on Windows: venv\Scripts\activate
pip install -r requirements.txt
```

### 2. Run EDA (optional, generates plots)
```bash
python src/eda.py
```

### 3. Train the model
```bash
python src/train.py
```
This will:
- download the dataset via `kagglehub`
- clean and preprocess the data
- balance classes with SMOTE
- train and compare Decision Tree, Random Forest, and XGBoost
- save the best model to `models/customer_churn.pkl`
- save the encoders to `models/encoders.pkl`

### 4. Predict on a new customer
```bash
python src/predict.py
```
Edit the `new_data` dictionary in `src/predict.py` to test different customer profiles, or import `predict_churn()` into your own script/app.
